# Style Guide

本文件定义跨语言、跨工程的代码风格约束。语言专项规范由 `.claude/rules/*` 中的具体规则补充。

## 通用风格

- 优先清晰、稳定、可维护的实现，而不是炫技。
- 命名必须表达业务语义，不使用临时缩写和无意义缩写。
- 注释写“为什么”，不重复代码字面意思。
- 单次改动优先最小范围闭环，不顺手重写无关模块。
- 新增结构前优先复用现有结构、组件和工具。

## 文件与目录

- 文件和目录命名遵循项目现有风格，不混用多套命名体系。
- 同类对象放入稳定目录，不随意创建“misc”“temp”“util2”之类目录。
- 公共目录只放真正公共的内容。

## 前端风格

- 页面、区块、组件、hooks、services 职责清晰。
- 页面先处理结构，再接数据，再补状态。
- 组件封装遵守：
  - 基础组件
  - 领域组件
  - 页面区块组件
  - 页面组件
- 避免把 API 请求、页面级状态和复杂业务编排塞进纯展示组件。

## 后端风格

- controller / service / repository 分层清晰。
- 业务规则不散落在 controller、entity 或 mapper 中。
- 参数校验、权限、事务、异常响应要明确。
- DTO / VO / Entity 分层清楚，不直接暴露持久化对象。

## SQL 风格

- SQL 优先可读性和执行计划稳定性。
- 不写无条件批量更新或删除。
- 不在业务层之外偷放权限控制逻辑。
- 表设计、索引设计、迁移说明必须一起考虑。

## 交付风格

- 最终说明必须包含改动内容、验证结果和剩余风险。
- 无法验证时要明确卡点和未验证范围。
- 不提交临时 mock、debug、console 或 TODO，除非任务文档允许。
