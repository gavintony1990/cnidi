# Go Backend Rules

本规则用于 Go 后端工程的实现与评审，补足 `CLAUDE.md` 中对工程边界、接口契约、验证和交付的约束。

## 工作前检查

开始实现前必须先确认：

- 已阅读 `CLAUDE.md`、`.claude/claude.md`、当前任务文档。
- 如涉及接口，已阅读 `.claude/docs/api-contract.md`。
- 已明确需求归属是否只影响后端，还是同时影响前端和后端。
- 已明确当前项目实际使用的 Web 框架、ORM、迁移方式和测试方式。

## 目录与分层

推荐结构：

```txt
cmd/
internal/
  api/
  service/
  repository/
  model/
  middleware/
  config/
pkg/
tests/
```

职责边界：

- `api` / `handler`：只做 HTTP/gRPC 适配、参数解析、响应返回。
- `service`：处理业务编排、权限决策、事务边界。
- `repository`：只做数据访问，不写业务规则。
- `model`：领域对象、DTO、持久化对象分层清晰。
- `middleware`：鉴权、日志、trace、统一错误转换。
- `config`：配置加载和外部依赖初始化。
- `pkg`：确实需要复用的通用能力，不放业务逻辑。

## 编码规则

- handler 不直接操作数据库。
- service 不拼接 HTTP 响应。
- repository 不返回框架私有错误给上层，必要时转换为可识别错误。
- 所有外部调用都必须传递 `context.Context`。
- 禁止随意引入全局变量保存请求态信息。
- 优先写清晰的结构和错误路径，不为了“简洁”隐藏行为。
- 接口返回使用 DTO/VO，不直接暴露数据库模型。

## 校验与错误处理

- 输入校验使用 `validator` 或框架 binding，校验失败返回结构化错误。
- 业务错误定义为稳定错误码，不直接返回底层数据库或第三方错误。
- middleware 负责把业务错误转换为统一响应：

```json
{
  "error": {
    "code": "FORBIDDEN",
    "message": "Permission denied",
    "details": [],
    "requestId": "req_123"
  }
}
```

- 关键路径必须记录请求标识、用户标识、关键参数摘要和失败原因。

## 数据与事务

- 事务边界放在 service 层。
- 数据库变更必须显式说明迁移影响和回滚方式。
- 写操作要考虑幂等、唯一约束、重复提交和并发安全。
- 查询接口要明确分页、排序和筛选边界，避免无界查询。
- 避免 N+1 查询、连接泄漏和未关闭事务。

## 安全与权限

- 后端是最终权限边界，不能依赖前端隐藏按钮。
- 所有写操作接口都要明确鉴权要求。
- 敏感操作建议记录审计日志。
- 不直接暴露堆栈、SQL、内部异常细节。

## 测试与验证

至少按改动范围执行：

```bash
go test ./...
go vet ./...
go build ./...
```

涉及并发或共享状态时增加：

```bash
go test -race ./...
```

如无法运行，必须明确：

- 卡点是什么
- 受哪些环境因素影响
- 未验证风险是什么

## 交付要求

最终说明必须包含：

- 改动内容
- 涉及模块
- 接口契约是否变更
- 验证命令与结果
- 剩余风险
