# SQL General Rules

本文件定义跨语言、跨后端框架通用的数据库表设计与 SQL 实现规则。Java、Go、Python 等后端在实现数据访问时都必须先遵守本文件，再结合具体数据库规则目录执行。

## 适用范围

适用于：

- 表结构设计
- 建表语句
- 索引设计
- CRUD SQL
- 分页、筛选、排序查询
- 事务型写操作
- 审计字段与逻辑删除
- 迁移脚本与回滚说明

## 工作前检查

开始设计表或写 SQL 前必须确认：

- 已明确业务实体、主键策略、查询模式和写入模式。
- 已明确当前使用的数据库类型。
- 已明确是否涉及权限、审计、逻辑删除、唯一约束和历史兼容。
- 已确认接口契约和数据模型是否已经落在 `.claude/docs/api-contract.md` 或任务文档中。

## 表设计原则

- 一张表只服务一个清晰的业务实体或关系，不混装多种业务语义。
- 表名、字段名必须有稳定业务含义，不使用临时缩写和无意义命名。
- 所有表建议包含基础审计字段，除非明确说明不需要：
  - `id`
  - `created_at`
  - `created_by`（如适用）
  - `updated_at`
  - `updated_by`（如适用）
- 逻辑删除表建议显式包含删除标记和必要审计字段，而不是隐式约定。
- 枚举状态字段必须有明确状态集合，不允许多个字段共同拼出模糊状态。

## 主键与唯一约束

- 主键策略要在业务层可解释，不要随意混用自增、UUID、雪花 ID。
- 对自然唯一业务键必须建立唯一约束，不能只靠代码判断。
- 联合唯一约束优先用于表达真实业务唯一性，而不是靠应用层兜底。

## 索引规则

- 所有高频查询都必须先从查询条件反推索引。
- 索引为查询服务，不为“看起来规范”而堆索引。
- 对分页、筛选、排序场景，要明确联合索引顺序。
- 不在低选择性字段上盲目建立单列索引。
- 新增索引时要说明服务于哪些具体查询。

## SQL 编写规则

- 查询字段必须按实际需要选择，避免无意义 `select *`。
- 分页查询必须显式限制返回行数。
- 排序必须稳定，避免分页结果漂移。
- 批量更新、删除必须明确条件，禁止无条件全表写操作。
- 条件拼接必须可读、可验证，避免难以维护的动态 SQL。
- SQL 实现必须兼顾可读性和执行计划，不为了“短”牺牲稳定性。

## 事务与并发

- 事务边界应由业务层明确控制，SQL 层只负责数据操作。
- 重要状态流转必须评估并发更新、重复提交和幂等问题。
- 涉及库存、余额、配额、状态抢占等场景时，必须明确锁策略或版本控制策略。
- 批量写操作要明确失败处理和回滚策略。

## 逻辑删除与历史数据

- 逻辑删除仅在业务确实需要历史保留时使用。
- 使用逻辑删除后，查询语句必须统一处理删除标记。
- 唯一约束、恢复数据、重复创建等场景必须提前设计。
- 历史表、归档表、流水表不要混用主业务表语义。

## 审计与安全

- 管理后台敏感数据变更建议有审计字段或审计日志支撑。
- 禁止把权限控制仅放在 SQL where 条件拼接上，而没有业务层明确约束。
- 不在 SQL 中泄露敏感明文处理逻辑。

## 迁移与变更

- 建表、改表、索引变更都必须通过迁移脚本管理。
- 变更必须说明：
  - 影响表
  - 影响字段
  - 兼容性影响
  - 回滚方式
- 禁止直接修改线上表结构而不留迁移记录。

## 交付要求

涉及数据库变更时，最终说明至少包含：

- 表或字段变更内容
- 索引变更内容
- 受影响 SQL 或查询场景
- 迁移与回滚说明
- 验证方式
