# Java Algorithm and Design Pattern Rules

本文件定义 Java 后端在算法设计和设计模式使用上的约束，适用于 `bootServiceWorkSpace`。

目标不是为了“展示模式”而堆砌抽象，而是让后端代码在复杂度上升时仍然保持可维护、可测试、可演进。

## 总体原则

- 优先用简单、直接、可读的实现解决问题。
- 只有在重复逻辑、复杂分支、状态流转、对象构建或扩展点明确存在时，才引入设计模式。
- 算法选择优先服务于正确性、复杂度和可维护性，不为了炫技引入不必要的数据结构。
- 设计模式是工程手段，不是目标；如果引入模式后反而增加理解成本，应回退到更直接的实现。

## 算法规则

### 复杂度意识

- 编写查询、聚合、去重、排序、匹配类逻辑时，必须明确时间复杂度和空间复杂度。
- 默认避免无界嵌套循环；当数据规模增长可能明显放大成本时，优先考虑索引结构、哈希结构或预处理。
- 对列表扫描、批量处理、导入导出、统计汇总类逻辑，必须评估是否会退化为 O(n²) 或更差复杂度。

### 数据结构选择

- 唯一性判断优先使用 `Set`，不要在 `List` 上反复 `contains`。
- 键值检索优先使用 `Map`，不要用顺序遍历模拟字典查询。
- 保序要求明确时再使用 `LinkedHashMap` / `LinkedHashSet`。
- 需要排序时优先评估是数据库排序还是内存排序，不要默认把大量数据拉到内存后再排序。

### 后端场景约束

- 能在数据库层完成筛选、分页、排序、聚合的，不要无原则搬到 Java 内存层处理。
- 能通过 SQL / 索引解决的性能问题，不先用复杂 Java 算法掩盖数据库设计问题。
- 批量操作优先做分批处理、游标或分页处理，不一次性把大数据集全部读入内存。
- 状态机、权限判定、规则匹配类逻辑必须显式整理规则顺序，不允许散落在多个 if/else 中不可追踪。

### 可读性规则

- 不使用晦涩缩写命名中间变量，例如 `tmp1`、`res2`、`map3`。
- 算法中关键步骤必须能从方法名、变量名或短注释中看出意图。
- 复杂转换链不应过度依赖多层 Stream 嵌套；当链条影响可读性时，拆回普通流程代码。

## 设计模式规则

### 使用前提

引入模式前必须回答三个问题：

1. 现在的代码痛点是什么。
2. 这个模式具体消除哪个复杂度。
3. 不用这个模式会产生什么实际维护问题。

如果答不清，就不要引入。

### 推荐模式及适用场景

#### Strategy（策略模式）

适用于：

- 多种规则匹配
- 多种计费/计算方式
- 多种导入导出处理逻辑
- 多种渠道/来源差异化处理

规则：

- 使用策略模式时，策略选择条件必须集中管理。
- 不允许一部分分支在策略工厂里，一部分分支还散落在业务代码里。

#### Factory / Simple Factory（工厂模式）

适用于：

- 对象创建过程复杂
- 根据类型生成不同实现
- 需要屏蔽创建细节

规则：

- 工厂只负责创建，不负责业务判断。
- 工厂类不要逐渐演化成“全能调度中心”。

#### Template Method（模板方法）

适用于：

- 多个流程骨架一致，但少量步骤不同
- 审批流、同步流、导出流这类有固定主流程的场景

规则：

- 模板类必须稳定，变化点通过受控钩子方法暴露。
- 不要把每一步都开放成可重写，否则模板会失去约束价值。

#### Builder（建造者模式）

适用于：

- 对象字段较多
- 创建过程分步完成
- 需要提升构造可读性

规则：

- 只在对象构造明显复杂时使用。
- DTO / Entity 简单赋值场景不要机械引入 Builder。

#### Chain of Responsibility（责任链）

适用于：

- 校验链
- 过滤链
- 规则逐步匹配
- 审批节点前置校验

规则：

- 责任链必须明确顺序、停止条件和失败返回。
- 不允许责任链节点之间隐式依赖共享可变状态而没有约束。

#### State（状态模式）

适用于：

- 明确状态流转模型
- 不同状态下行为不同
- 状态切换规则复杂

规则：

- 状态模式适合复杂状态机；简单两三个状态不要过度设计。
- 状态流转必须有单独的可读表达，不允许边界条件散落。

### 谨慎使用的模式

#### Singleton

- Spring 管理的 Bean 默认就是单例语义，通常不需要手写 Singleton。
- 禁止为了“全局访问方便”手写不受控单例。

#### Observer / Event 模式

- 只有当解耦收益明显时使用。
- 事件链必须可追踪，不允许把关键业务副作用埋到看不见的事件监听中。

#### 过度抽象接口

- 只有存在多实现或明确扩展点时再抽接口。
- 不要把每个 service 都先写成接口 + 单实现，只为了形式统一。

## Spring Boot 场景下的模式落点

- Controller 层避免模式堆叠，保持薄。
- Service / application 层是策略、模板、状态、责任链等模式的主要落点。
- Repository / mapper 层保持数据访问职责，不承载设计模式驱动的业务编排。
- Config 层只负责装配，不把业务判断写进 Spring 配置类。

## 禁止行为

- 不允许为了“显得高级”引入设计模式。
- 不允许在业务尚未稳定时过早抽象多层扩展点。
- 不允许用复杂算法掩盖数据库设计或 SQL 设计问题。
- 不允许把关键业务规则隐藏在难以追踪的反射、动态代理或过度泛型中。
- 不允许把简单 CRUD 场景包装成多层模式拼图。

## 落地要求

当代码中引入明显算法优化或设计模式时，提交说明或代码注释中至少要能说明：

- 解决的具体问题
- 为什么当前复杂度值得这样设计
- 如果是模式，模式的边界和职责
- 如果是算法优化，复杂度改进点和适用范围
