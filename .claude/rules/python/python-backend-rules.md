# Python Backend Rules

本文件定义 Python 后端实现规则，适用于项目中存在 Python 服务或用户明确要求 Python 方案的场景。

## 工作前检查

开始实现前必须确认：

- 已阅读 `CLAUDE.md`、`.claude/claude.md`、当前任务文档。
- 如涉及接口，已阅读 `.claude/docs/api-contract.md`。
- 已明确当前需求是否真实落在 Python 工程中，而不是技术选型讨论。

## 工程与分层

推荐结构：

```txt
app/
  api/
  services/
  repositories/
  models/
  schemas/
  tasks/
tests/
```

职责要求：

- `api`：路由、参数解析、响应返回。
- `services`：业务编排、权限、事务边界。
- `repositories`：数据访问。
- `models`：ORM / 持久化对象。
- `schemas`：输入输出模型。
- `tasks`：异步任务。

## 编码规则

- 路由层不写复杂业务逻辑。
- 输入输出使用明确 schema，不直接返回 ORM 模型。
- 异步任务要考虑幂等、重试和失败补偿。
- 类型提示优先，避免无边界动态字典贯穿多层。
- 不把临时脚本式写法带进正式服务代码。

## 校验、异常与日志

- 参数校验必须统一。
- 业务异常使用统一错误结构，不直接把底层异常暴露给前端。
- 日志记录请求标识、用户标识、关键对象标识和失败原因。
- 不输出敏感信息。

## 测试与验证

至少按改动范围执行：

- 单元测试
- 接口测试
- 构建 / 静态检查（如项目存在）

如无法运行，必须说明未验证范围和剩余风险。

## 交付要求

最终说明必须包含：

- 改动内容
- 涉及模块
- 接口 / 数据影响
- 验证结果
- 剩余风险
